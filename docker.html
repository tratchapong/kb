MySQL stores date values using specific data types designed to handle temporal information like dates and times. Here's an overview of how it stores and represents date values:

📅 Common MySQL Date and Time Data Types

Data Type Format Description
DATE 'YYYY-MM-DD' Stores only the date (year, month, day)
DATETIME 'YYYY-MM-DD HH:MM:SS' Stores date and time
TIMESTAMP 'YYYY-MM-DD HH:MM:SS' Similar to DATETIME, but also stores time zone-converted values
TIME 'HH:MM:SS' Stores only time
YEAR 'YYYY' or YY Stores only year (as 4-digit or 2-digit, depending on version)

🔧 How MySQL Stores DATE Internally

  • Internally, the DATE type is stored in 3 bytes.

  • It contains:

    • 1 byte for the year (offset from 1900 or encoded in some form)

    • 1 byte for the month

    • 1 byte for the day

So for example:

sql
CREATE TABLE events (
    id INT,
    event_date DATE
);

INSERT INTO events VALUES (1, '2025-05-28');

MySQL will store '2025-05-28' in a compact 3-byte format internally but display it in the standard 'YYYY-MM-DD' format when queried.

🧠 Notes

  • MySQL automatically validates date formats. Invalid dates (e.g., '2025-02-30') will either produce an error or be stored as '0000-00-00' depending on the SQL mode.

  • You can manipulate date values using functions like NOW(), CURDATE(), DATE_ADD(), DATEDIFF(), etc.

Does it store date as number of milliseconds since January 1, 1970?
ChatGPT said:

No, MySQL does not store DATE, DATETIME, or TIMESTAMP as the number of milliseconds since January 1, 1970 (Unix epoch), unlike systems like Unix/Linux timestamps or some NoSQL databases.

Here’s how it actually works:

❌ Not Stored as Milliseconds Since 1970

  • MySQL does not internally represent date/time as an integer or floating point number of milliseconds since the epoch.

  • There’s no automatic epoch-based timestamp system in DATE or DATETIME.

✅ How MySQL Actually Stores Date and Time

1. DATE

  • Stored in 3 bytes.

  • Each part (year, month, day) is encoded directly.

  • No relation to the Unix epoch.

2. DATETIME

  • Stored in 5 to 8 bytes (depending on fractional second precision).

  • Encoded as a combination of date and time fields, not a single integer.

  • Independent of time zones.

3. TIMESTAMP

  • This is the only temporal type that is logically tied to Unix time.

  • Internally stored as seconds since '1970-01-01 00:00:01' UTC, but still not directly as milliseconds.

  • Stored in 4 bytes (or more if fractional seconds are used).

  • When inserted or retrieved, values are automatically converted between UTC and the current time zone.

Summary

Data Type Epoch-Based? Internal Format
DATE ❌ No Encoded as Y-M-D
DATETIME ❌ No Encoded as Y-M-D H:M:S
TIMESTAMP ✅ Yes Stored as seconds since epoch

If you're looking for millisecond-precision timestamps, you’ll have to use either:

  • A BIGINT field and store UNIX_TIMESTAMP(NOW(3)) * 1000

  • Or use DATETIME(3) or TIMESTAMP(3) for microsecond storage (MySQL 5.6+)

Does PostGresSQL use the same way to store Date?
ChatGPT said:

PostgreSQL does not use the same method as MySQL to store DATE values. While both systems store dates in compact formats internally, PostgreSQL uses a different internal representation, which in some ways is closer to Unix time than MySQL's DATE type — but still not exactly milliseconds since epoch.

📅 PostgreSQL DATE Storage

🔧 Internally:

  • PostgreSQL stores DATE values as integers.

  • Specifically, it's the number of days since 2000-01-01 (the PostgreSQL date epoch).

  • This integer format makes arithmetic (e.g., date subtraction) efficient.

Example:

  • '2000-01-01' is stored as 0

  • '2000-01-02' is 1

  • '1999-12-31' is -1

So internally, PostgreSQL stores:

text
DATE = Integer (days from 2000-01-01)

🕒 PostgreSQL TIMESTAMP and TIMESTAMPTZ

TIMESTAMP WITHOUT TIME ZONE

  • Stored as a 64-bit floating point number or integer (depending on build).

  • Represents microseconds since 2000-01-01 00:00:00.

TIMESTAMP WITH TIME ZONE (TIMESTAMPTZ)

  • Also stored as microseconds since 2000-01-01 UTC.

  • Time zone adjustments are done on input/output, not storage.

So while it's not stored as milliseconds since 1970, PostgreSQL's TIMESTAMP is closer to the Unix timestamp idea than MySQL’s DATETIME.

🧠 Summary: PostgreSQL vs MySQL Date Storage

Feature MySQL DATE PostgreSQL DATE
Internal format Encoded Y/M/D Integer (days since 2000-01-01)
Epoch used N/A 2000-01-01
Storage size 3 bytes 4 bytes (int)
Arithmetic ease Less direct Very efficient (just integers)
Epoch-based? ❌ No ✅ Yes (but for days, not time)

Concept MySQL PostgreSQL Equivalent

  • If you migrate a MySQL database to PostgreSQL, convert MySQL DATETIME to PostgreSQL TIMESTAMP WITHOUT TIME ZONE.

  • PostgreSQL is stricter and clearer about time zone handling, whereas MySQL's DATETIME has no time zone behavior at all.

Yes — if your app involves a lot of complex date and time manipulation, PostgreSQL is generally the better choice over MySQL. Here's why:

✅ Why PostgreSQL Excels at Date/Time Manipulation

  1. Rich Set of Built-in Date/Time Functions

    • PostgreSQL has more comprehensive and standards-compliant date/time functions than MySQL.

    • Examples: AGE(), JUSTIFY_DAYS(), DATE_TRUNC(), INTERVAL, EXTRACT(), etc.

  2. Powerful Interval Support

    • PostgreSQL supports a true INTERVAL type with precise handling of months, days, hours, etc.

    • Makes things like "add 3 months and 10 days" very straightforward.

  3. Time Zone Intelligence

    • TIMESTAMP WITH TIME ZONE is actually time zone-aware (unlike MySQL’s misleadingly named TIMESTAMP).

    • You can easily convert, store, and compare across time zones.

  4. Mathematical Date Arithmetic

    • Since DATE is stored as an integer (days from 2000-01-01), operations like date difference, addition, and subtraction are fast and simple.

  5. Better Standards Compliance

    • PostgreSQL adheres more closely to the SQL standard, making its behavior more predictable and portable.

⚠️ MySQL Limitations

  • Weaker INTERVAL handling and less expressive syntax.

  • Fewer built-in date/time manipulation functions.

  • Historically inconsistent behavior in DATETIME vs TIMESTAMP.

  • Time zones can be confusing or inconsistently applied unless explicitly configured.
